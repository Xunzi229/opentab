# OpenTab 冻结需求

本文档记录已经确认、后续默认不应随意改变的开发要求。  
如需调整，必须先更新本文档，再修改实现。  
页面设计在未明确指定时，应持续朝更整洁、更统一的方向优化，不保留明显重复结构。

## 1. 分组规则

- 分组名称不能为空。
- 分组名称不能重复。
- 重名判断按去首尾空格后的名称进行。
- 默认分组 `未分组` 必须存在。
- 默认分组不能删除。
- 删除自定义分组时，分组内路由必须自动移动到 `未分组`。

## 2. 路由规则

- 收藏记录必须保存完整、可访问的 URL。
- 管理页中的路由必须可以直接点击打开。
- 点击路由默认使用新标签页打开。
- 管理页中的路由展示采用单条列表形式，不再使用小卡片栅格。
- 每条路由都必须支持这些常用操作：
  - 打开
  - 删除
  - 移动到其他分组
  - 编辑
  - 星标
- 必须支持批量打开分组内全部地址。
- `chrome://` 等非 `http(s)` 页面不在收藏范围内。

## 3. 管理页规则

- 管理页面必须是独立页面。
- 独立管理页必须使用 `manager.html`。
- `sidepanel.html` 只用于 Chrome Side Panel，不再充当独立管理页。
- Popup 中必须提供明确的“打开管理页面”按钮。
- 通过 Popup 打开管理页时，必须在原来的浏览器窗口中新建固定标签页。
- 当管理页已打开时，应优先聚焦已有标签页，并保持固定标签页状态，而不是重复新开。
- 点击扩展图标时，不允许默认把 Side Panel 挂到当前页面右侧。

## 4. 最近访问规则

- 最近访问最多保留并展示 10 条记录。
- 最近访问按最近时间倒序排列。
- 最近访问页面避免重复标题、重复说明和重复容器，只保留一个主内容块。

## 5. 数据规则

- 当前版本默认使用 `chrome.storage.local`。
- 本地收藏、分组、标签、访问记录和设置必须持久化。
- 后续接入 Chrome Sync 时，不得破坏本地可用性。

## 6. 备份规则

- 导入导出不得直接使用裸 JSON 文件作为用户备份格式。
- 备份文件必须使用统一的加码 / 解码流程，至少避免普通用户直接编辑原始结构。
- 备份文件扩展名固定为 `.opentab`。
- 导出文件名固定格式为 `backup_时间戳.opentab`。
- 导入和导出页面、以及设置页中的导入导出入口，必须使用同一套备份编解码逻辑。

## 7. 版本管理规则

- 插件版本必须纳入工程化管理。
- 每次执行打包命令时，必须自动递增插件版本号。
- `manifest.json` 中的扩展版本必须与项目版本保持一致。
- 不允许依赖手工修改版本号完成发版。

## 8. 开发方向约束

以下方向已经固定：

1. 管理效率优先于装饰性布局。
2. 独立管理页优先于 Side Panel 形态。
3. 路由“可打开”是刚性要求，不是增强项。
4. 分组唯一性是刚性要求，不是提示性建议。
5. 打包自动递增版本是刚性要求，不是人工流程。
6. 列表式路由管理优先于卡片式展示。
7. 批量操作能力属于核心管理能力。
